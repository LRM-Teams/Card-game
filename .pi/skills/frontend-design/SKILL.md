# Frontend Design（斗地主团队版）

这是团队共享的设计纪律 skill。任何 agent 改 UI 之前先按这里走一遍，改完截图复验再按这里走一遍。
目标：让界面**一眼看懂在干嘛、信息不重复、有清晰的主次**，而不是“功能都在但视觉混乱”。

## 0. 一条铁律：先定“这一屏的一件事”

动手前先回答：**这一屏用户最重要的一个动作是什么？**
斗地主牌桌的答案通常是「看清当前轮到谁 + 我要出什么牌」。其它信息（倍数、倒计时、身份、历史）都是配角，不能和主信息抢视觉权重。
如果一句话说不清这屏的“一件事”，就先收敛信息，再开始排版。

## 1. 视觉层级审计（必做，尤其当有人说“乱/重复”时）

逐项列出屏幕上的信息块，按“用户当下是否需要”分级：

- **主信息**（占最大视觉权重，居中/高亮）：当前出牌「谁出了什么牌」、我的手牌、出牌按钮。
- **次信息**（中等权重，固定位置）：当前轮到谁、倒计时、倍数。
- **背景信息**（弱权重，常驻不抢戏）：身份（地主/农民）、座位头像、剩余牌数。

红线（必须消除的“混乱/重复”）：
- **同一信息出现两次**：例如身份既用 SVG 徽章又用文字牌叠在一起 → 只保留一种（徽章优先，文字仅作极简补充）。
- **多个同权重的彩色块**同时竞争：倒计时、倍数、身份、按钮如果都是亮色 → 只让“出牌”这一处最亮，其它降饱和/收到角落。
- **标签堆叠**：玩家名、座位号、身份、牌型同时贴在一个区域 → 合并成一行强信息（`{玩家名} 出了 {牌型}`），删掉冗余副标签。
- **散落的状态点**：把倒计时/倍数/轮次合并到一个角落小组件，不要三点散开。

## 2. 先出设计计划，再写代码（brainstorm → tokens → plan → critique → build → critique）

两轮工作法：

**第一轮 · 计划（不写业务代码）**
- **调色板**：4–6 个命名 hex，走 token（见 `docs/doudizhu-design-tokens.md` 的 `--ddz-*`）。牌桌台呢绿为基底，金色为唯一强调色，红/黑为牌面层级。
- **字体**：展示字体（牌型/标题）+ 正文字体（信息），定好字号阶梯。
- **布局**：一句话描述 + 一个 ASCII wireframe，说明主信息放哪、次信息放哪、留白在哪。
- **签名**：这一屏让人记住的一个独特元素（如桌中央的桌徽/出牌高亮带）。

**第二轮 · 复盘计划再动手**
把计划对着“这一屏的一件事”和“层级审计红线”检查一遍：
- 有没有信息重复？有没有同权重彩色块打架？
- 配色是不是又掉回默认深绿模板？需不需要暖光/层次拉开桌面与背景？
- 卡牌/图标尺寸是否挤占了留白？
确认无“模板感/混乱感”后，再写代码，且**每一个颜色和尺寸都从 token 派生**。

## 3. 斗地主专属约定

- **牌桌是主视觉**：桌面（台呢绿 + 厚桌沿 rail）必须和页面背景（暖色“房间”）拉开明显落差，让人看到“一张桌子摆在房间里”，而不是“一块绿色大面板”。
- **卡牌层级**：手牌是底部主焦点、轻微扇形；对手牌更小；中央出牌区是强信息带。
- **身份一眼可分**：地主/农民用图标区分（地主=皇冠，农民=麦穗），只保留徽章一种形式，不要 SVG+文字双叠。
- **图标不手搓**：用开源可商用图标库（Iconify 等，如 `mdi:crown` / `mdi:wheat`），不自造图标；记录来源 + license。
- **结束态清晰**：结算页有明确的“再来一局”强主按钮，胜负面板层级清楚。
- **动画服务信息**：出牌、发牌、选中、炸弹等动效是为了让“谁干了什么”更清楚；散乱的常驻特效会显得 AI 味重，克制使用。

## 4. 复验清单（贴截图前自查）

- [ ] 这一屏的“一件事”是否最显眼？
- [ ] 有没有同一信息出现两次（徽章+文字、座位号+名字等）？
- [ ] 同时有几个亮色块在抢？是否只有“出牌”最亮？
- [ ] 卡牌/图标是否太大、挤掉留白？
- [ ] 桌面和背景是否拉开了落差（像一张桌子）？
- [ ] 所有颜色/尺寸是否走 token，没有硬编码？
- [ ] 移动端宽度下是否仍然清晰不挤？
